# 테스트 시트 작성 가이드

> 변경사항/기능에 대한 테스트 케이스를 빠짐없이 도출하기 위한 절차. happy path만이 아니라
> worst/edge/통합 경로, 그리고 **쉽게 누락되는 축**(렌더 입력·극단 길이·브라우저 엔진·뷰포트)을
> 케이스 설계 단계에서 선제 포착한다.

---

## 1. 절차

1. **변경면(change surface) 분석** — diff/PR에서 실제로 바뀌는 입력·출력·상태·부수효과를 나열.
2. **케이스 도출** — 각 변경면마다 정상/실패/경계 케이스를 작성. 케이스에 ID(C#/B#)·기대값·결과 칸.
3. **커버리지 매핑** — 각 케이스를 단위(vitest)·E2E(Playwright)·수동(curl/브라우저) 중 어디서 검증할지.
4. **실행 후 결과 기록** — PASS/FAIL + 근거(실측값). 추측 금지([EMPIRICAL_VERIFICATION.md](./EMPIRICAL_VERIFICATION.md)).

산출물 형식 예시(curl C#, 브라우저 B#)는 [RELEASE_VERIFICATION.md](./RELEASE_VERIFICATION.md) 참조.

---

## 2. 누락 방지 체크리스트 (가장 중요)

아래는 실제로 **사용자 피드백으로 뒤늦게 드러난 누락**들이다. 처음부터 이 질문을 던진다.

### 2.1 렌더 입력이 사용자/외부 데이터인가?
- 입력이 **마크다운/HTML/리치 텍스트**라면 → 렌더링 케이스 필수(굵게/리스트/코드/링크/이미지).
  - *사례*: 공지 팝업이 DB의 마크다운을 렌더하는데 마크다운 렌더 케이스가 없었음.
- 입력이 외부 API 응답이라면 → 누락 필드/null/에러 응답/스키마 변경 케이스.

### 2.2 길이·수량의 극단값은?
- **매우 긴 입력**(긴 본문, 긴 목록, 큰 숫자)에서 레이아웃이 깨지거나 사용성을 해치지 않는가?
  - *사례*: 공지 본문이 길어지면 모달이 화면을 넘겨 푸터 버튼이 밀려남 → 본문만 스크롤·푸터 고정 검증 필요.
- 0개/1개/N개 경계, 빈 문자열, 최대 길이.

### 2.3 브라우저 엔진 차이는?
- **Chrome(Blink)만 보고 Safari(WebKit)를 빠뜨리지 않는가?** 스케줄링·CSS·포커스 거동이 다르다.
  - *사례*: `startTransition` starvation이 Chrome에서만 재현, Safari는 정상.
- 매트릭스는 [MULTI_ENV_TESTING.md](./MULTI_ENV_TESTING.md) 표준을 따른다.

### 2.4 뷰포트(모바일)는?
- 데스크톱만 보고 **모바일**을 빠뜨리지 않는가? `dvh`(주소창)·터치·가로 오버플로우·시트 거동.

### 2.5 캐싱·상태 영속은?
- Redis/Next 캐시가 끼면 → 캐시 히트/미스/무효화 케이스. 실제 Redis로 검증([QA_ENV_SETUP.md](./QA_ENV_SETUP.md)).
- localStorage/cookie 영속 → 새로고침/재접속 후 상태 유지·소멸 케이스.

### 2.6 접근성(a11y)은?
- 키보드 단독 사용자가 모든 동작에 도달 가능한가?(포커스 트랩, 스크롤 영역 `tabIndex`, ARIA)
  - *사례*: 스크롤 본문에 포커스 가능한 자손이 없으면 키보드로 스크롤 불가(WCAG 2.1.1).

## 3. 케이스 설계 원칙

- **결정적 매처**: 값이 고정이면 정확히 단언한다. `toContain('max-h-')`(아무 max-h-* 통과)가 아니라
  `toContain('max-h-[85dvh]')`. (MISTAKES.md Tests §13)
- **회귀 가드 불변식**: 단위 테스트는 jsdom이 측정 못 하는 시각 동작 대신 **구조 불변식**을 단언하고,
  시각 동작은 E2E로 보낸다. 단, "E2E로 검증된다"는 주석은 **실제 커밋된 E2E가 있을 때만** 적는다
  (없는 검증을 인용하는 false WHY 주석 금지 — MISTAKES.md §15.6).
- **worst/edge/통합**: 정상 1개당 실패·경계·통합을 함께. 통합은 슬라이스 경계를 넘는 흐름.
