# Personalized Analysis by Position Bucket (Subsystem C) — Change Scope & Empirical Test Cases

- **Date**: 2026-07-18
- **Type**: CHANGE-SCOPE SPEC + empirical (실증) TEST PLAN — NOT a design doc. Companion to `2026-07-18-personalized-analysis-position-bucket-design.md`.
- **Feature**: Personalize the technical/chart AI analysis to a logged-in member's unrealized P&L on the symbol, via a coarse server-derived "position bucket", without shattering the shared analysis cache.
- **Repos / branches**:
  - `siglens` — `feat/personalized-analysis` (stacked on `feat/portfolio-position-viz`)
  - `siglens-core` — `feat/position-bucket-analysis` (off `main`), released as **0.37.0**
- **Purpose of this doc**: drive manual/empirical verification via `curl` + a real browser against a production-like build. Every test case below has a concrete, checkable observable.

|----|-----------|-----------------------|
| INV-1 | **No-bucket cache key is byte-identical to legacy.** `positionKeySuffix(undefined) === ''`; `buildAnalysisCacheKey` appends nothing. Every anonymous/free/no-holding request reads/writes the exact pre-feature key. | `positionKeySuffix` returns `''` on `undefined`; core `config.test.ts` asserts byte-identity. |
| INV-2 | **`PROMPT_TEMPLATE_VERSION` unchanged (`p6`).** Bucketed requests land on a net-new `:pos=` namespace; no previously-cached entry becomes stale; no warm-cache cold-start (technical + fundamental + all SSR shells untouched). | C6; core `config.test.ts` asserts `p6`. |
| INV-3 | **SSR / ISR HTML is never personalized.** `peekAnalysisStatic` pins no-bucket, so crawled/ISR shells always render the shared base analysis. | INV-4 | **Free / anonymous callers are excluded.** `resolvePositionBucket` and `resolveHoldingPositionBucket` both hard-gate `tier === 'free'` / `userId === null` → no DB read, no bucket, `personalized: false`. | S2, S3. |
| INV-5 | **No new env var, no DB migration.** Reuses existing `portfolio_holdings` (Subsystem A) + the existing Redis-cached quote. No new secret required for build/runtime. | Diff introduces no `.env`/schema/migration files. |
| INV-6 | **Degrade-to-base on ANY failure.** Holding-read error, quote-read error, degenerate avg, unbucketable position → `undefined` → base analysis + `personalized: false`. Underlying submit never blocked. | S3 try/catch; TC-04. |
| INV-7 | **Badge honesty — server-authoritative, never over-claims.** Badge gates on the server `personalized` flag threaded through `isPersonalized`, NOT on holding existence. Deferred during polling; reset on error/symbol-nav; hidden for fallback/degraded. Under-claim (badge one round-trip late) is acceptable; over-claim is not. | S6, S7, S9; TC-05, TC-06. |
| INV-8 | **Bucket never joins the skill-sampling seed.** The same skills are sampled regardless of bucket — only the prompt hint + cache key differ. | C8 (seed unchanged); core `submitAnalysis.test.ts`. |

## 4. Known limitations / accepted tradeoffs (verify the mitigations, don't flag as bugs)

- **Bounded per-submit cost.** Every member-with-holding submit pays a DB read (`findByUserAndSymbol`) + a quote read (`getQuote`), even when the analysis itself is a pure cache hit. Mitigated by reusing the existing Redis-cached quote (`quote:<SYMBOL>`, session-aware TTL) — normally a warm hit, not a fresh FMP call — but it is a real added dependency. Free/anon skip both reads entirely.
- **Quote-vs-snapshot drift at bucket boundaries.** The bucket's current price comes from the quote cache, which can drift intraday from the analysis snapshot's own last-bar close (different TTLs/fetch times). Near a band edge the bucket may occasionally disagree by a small margin with the price the user sees in the snapshot. Accepted for a coarse ~5-band bucket.
- **Badge appears one round-trip after a personalized SSR seed would.** The SSR/ISR seed is always the shared no-bucket base (peek pinned no-bucket, INV-3), so a member's personalized result + badge only arrive after a CLIENT re-submit resolves. This is a deliberate **safe under-claim**: the badge is late rather than wrong, and never over-claims on top of a base analysis.
- **v1 scope = technical/chart analysis only.** The `:pos=` axis is composed only for the technical analysis key. Fundamental/overall keys are untouched (mirror later if extended).
- **Cardinality.** Analysis key namespace grows by ×(≤5 buckets + no-bucket) for members-with-holdings only. Bounded, no explosion — this is why bucketing (not raw avg) was chosen. Net-new personalized cache writes for members should be monitored.
