# E2E Tier 2 — Auth/Account Specs Implementation Plan (PR-B)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Checkbox steps.
> **Repo policy:** implementer commits in the worktree; final PR + merge via git-agent. Multi-line JSDoc allowed.
> **Hard lessons from Tier 1 — bake into EVERY task:** (1) §22 — add colocated unit tests for EVERY new factory/fake/dispatcher/helper (all branches), or the review bounces. (2) Hermetic — NO real Resend/OAuth/network under E2E. (3) `.env.local` masks missing secrets locally; CI has only `.env.e2e` — add ALL required secrets up front and verify a cold build + e2e run rely only on `.env.e2e`. (4) require-gate fakes to keep them out of prod bundle (config-level eslint override, NO inline disable). (5) Avoid import cycles (extract shared interfaces to their own file).

**Goal:** Add the 6 auth/account E2E specs (login, signup, password-reset, oauth, account-api-key, account-delete) + the hermetic infra they need (fake email dispatcher, fake OAuth adapter, authed storageState, env secrets), running against the real local Postgres+Redis.

**Architecture:** Auth/session/DB run for REAL (local Postgres via the existing postgres-js swap, Redis via SRH). Email + OAuth providers are faked under `E2E_TEST=1`: a fake email dispatcher writes the plain verification code / reset token to a debug Redis key (since the real token store only keeps a hash) so specs read it via `e2e/support/srhClient.ts`; a fake OAuth adapter returns a fixture profile for `code=e2e_*`. Account-page specs run authenticated via a Playwright storageState seeded from a DB user+session.

## Pre-flight facts (from investigation)
- Cookies: `siglens_session` (httpOnly UUID = `sessions.id`), `siglens_auth`='1' (client hint). `createAuthSession` inserts a `sessions` row (id, userId, expiresAt=+30d, createdAt).
- Email token store (`src/entities/email-token/api.ts`): Redis key `email_token:{purpose}:{email}` (purpose ∈ email_verification|password_reset), value stores a HASH (not the plain code) → specs CANNOT read the code; the fake dispatcher must expose it.
- Email dispatcher: `src/shared/email/dispatcher.ts` `createEmailDispatcher()` — gate here under E2E. (RESEND_API_KEY no-ops if unset; EMAIL_FROM defaults.)
- OAuth: `src/features/auth-oauth/lib/providers.ts` `getOAuthAdapter()` (gate here); state cookie `siglens_oauth_state` HMAC-signed with `OAUTH_STATE_HMAC_SECRET` (THROWS if unset, min 32 bytes); pending signup Redis `pending_oauth_signup:{token}`; callback `/api/auth/callback/[provider]`. Adapter `exchangeCodeForProfile(code)` → fake returns fixture profile for `e2e_*` codes.
- API-key encryption: `src/shared/db/tokenEncryption.ts` AES-256-GCM, env `LLM_API_KEY_ENCRYPTION_KEY` (32-byte hex, THROWS if unset). Table `userApiKeys`. Actions in `src/entities/api-key/actions/`.
- `NEXT_PUBLIC_SITE_URL` (or OAUTH_REDIRECT_BASE_URL) THROWS if unset (builds OAuth callback URI) → must = `http://localhost:4300`.
- Account routes protected by `src/proxy.ts` (redirect to /login if no session).
- Per-spec selectors: see the investigation report (login `이메일`/`비밀번호`/`로그인`; signup 3-phase `인증 코드 받기`/`코드 확인`/`가입 완료` + consent checkboxes; reset `재설정 링크 보내기`/`비밀번호 변경`; account-delete `계정 영구 삭제` + email-match; api-key `저장`/`등록됨`/`삭제`). Verify against real `ui/` components.

## Tasks 1–6: the specs (each: import from `../support/fixtures`; assert OUTCOMES; run green chromium; commit)
- [ ] **auth-login.spec** — valid login → authed + redirect to `next`/account; invalid creds → error message; reverse-guard (logged-in visiting /login → redirect /). (Login uses the seeded auth user.)
- [ ] **auth-signup.spec** — phase1 request code → read code from Redis (emailHelper) → phase2 verify → phase3 register (password+consents) → authed/redirect. Use a UNIQUE email per run (timestamp/worker) to avoid "already registered".
- [ ] **auth-password-reset.spec** — request reset (seeded user) → read token from Redis → open `/reset-password?email&token` → set new password → redirect `/login?password_reset=1`; (optional) new password logs in, old fails.
- [ ] **auth-oauth.spec** — drive `/api/auth/google/start?next=/account` → callback with `code=e2e_google_code`&state (the fake adapter returns the fixture) → consent page (new user) → finalize → authed; (optional) existing-user auto-login + email-conflict branch.
- [ ] **account-api-key.spec** (authed storageState) — /account → save Anthropic key → "등록됨" badge → delete → "미등록". (Real AES encryption with the env key.)
- [ ] **account-delete.spec** (authed storageState; use a DEDICATED throwaway user, not the shared storageState user, OR re-seed) — /account/delete → wrong email → disabled/mismatch; correct email → enabled → delete → redirect `/?account_deleted=1`; verify user gone from DB. NOTE isolation: deleting the storageState user breaks other specs — seed + auth a SEPARATE delete-only user/session for this spec.
